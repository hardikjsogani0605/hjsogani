02 Selected Work
CASE 01
Goldman Sachs
2025 — Present
AI · LLM · Governance

Production AI through enterprise governance

Shipping a regulatory classification engine: 400+ quarterly U.S. tax regulatory updates ingested by LLM, classified, and routed. ~90% reduction in manual processing effort.

The problem

Goldman Sachs tax operations ingests 400+ U.S. regulatory updates every quarter. Each one has to be read, classified by domain and urgency, and routed to the right workstream — historically a manual senior-analyst job consuming hours of expert judgment time on inputs that mostly weren’t the high-judgment ones. The obvious solution was AI. The non-obvious problem was shipping it inside a Tier-1 financial institution where AI governance, compliance review, and data-handling guardrails shift quarter to quarter.

How I framed it

Most AI program failures aren’t model failures — they’re handoff failures. So I scoped this less as an LLM problem and more as a governance and workflow problem with an LLM at the center. Three parallel tracks: the classification pipeline itself with engineering, the AI governance review and compliance sign-off path, and the human-in-the-loop checkpoints operations would actually trust at scale.

What I shipped
  • LLM-based classification and routing pipeline processing 400+ updates per quarter
  • AI governance review artifacts: model risk documentation, data-handling controls, evaluation criteria
  • Human-in-the-loop review interface designed around how analysts actually work, not a theoretical workflow
  • Evaluation harness so accuracy can be measured on each new quarterly batch — not just at launch
  • Stakeholder communication plan that gave 30+ VP/MD-level stakeholders ongoing visibility and confidence
Outcome

~90% reduction in manual processing effort. Pipeline live in production. Senior analyst capacity reclaimed for actual judgment work.

What I’d tell another TPM

In regulated enterprise AI, the model is the easy part. The hard part is designing the program so governance, ops, and engineering can all say yes to the same artifact — and so the system has an evaluation story when audit comes asking.

CASE 02
Goldman Sachs
2025 — Present
GenAI · Workflow

Using AI on the TPM bottleneck

AI-generated executive status reporting across 6 workstreams. Weekly synthesis time collapsed from ~4 hours to ~30 minutes. Audience: VPs, OSLs, and MDs.

The problem

Every week I was burning ~4 hours stitching together status from 6 workstreams — engineering, ops, vendor PMs, multiple geographies — into a coherent narrative for VPs, OSLs, and MDs. The bottleneck wasn’t gathering inputs. It was the synthesis: which signals matter for an MD, which escalations need to lead, what gets cut. Classic TPM tax.

How I framed it

Two design principles up front. One: AI should compress synthesis time, not replace the program manager’s judgment — the final memo has my name on it, not the model’s. Two: executives don’t trust hallucinated status. The system had to be grounded in source workstream updates, not free-generation.

What I shipped
  • Ingest pipeline pulling raw status from 6 workstream owners
  • Structured synthesis prompts producing executive-ready summaries in the format MDs actually consume — escalations first, then progress, then asks
  • Mandatory manual review and edit checkpoint before anything reaches a VP+
  • Quality bar tracked over time — accuracy and edit-rate as health metrics
Outcome

Weekly reporting collapsed from ~4 hours to ~30 minutes. Capacity reallocated to engineer 1:1s and decision memos that actually move the program.

What I’d tell another TPM

The best AI program to ship first is the one that solves your own bottleneck. You’re the user, the QA, and the credibility owner. If you’d ship it for yourself, you can probably ship it for the org.

CASE 03
Goldman Sachs
2025 — Present
Ambiguity · Migration

Navigating ambiguity at scale

Migrating program delivery from Jira to internal workflow platform. No existing playbook. 5+ global teams. 150+ legacy tickets liquidated. 98% data accuracy.

The problem

Goldman decided to migrate tax operations program delivery from Jira to an internal workflow platform. No precedent in the org. No existing playbook. 5+ global engineering and ops teams with in-flight programs, none of which could pause while the migration ran underneath them.

How I framed it

When teams deadlock on tooling, my move isn’t to argue. It’s to design the test that makes the right answer undeniable. So I broke this into three: standardize the intake schema across teams before migrating anything (so we weren’t porting chaos), map technical dependencies between in-flight programs to determine migration sequence, and author a cutover plan with explicit rollback criteria.

What I shipped
  • Jira-based program governance framework: intake schema, prioritization criteria, reporting dashboards
  • Cleanup pass that liquidated 150+ legacy tickets and lifted data accuracy to 98%
  • Dependency map across 5+ global engineering and ops teams
  • Cutover plan with rollback gates — now the reference standard for internal-platform delivery
  • Onboarding playbook so incoming team members don’t have to discover the system from cold
Outcome

Migration on track. Governance framework adopted as reference standard. 98% data accuracy. Compounded into onboarding speed for new team members.

What I’d tell another TPM

In ambiguity, the most valuable artifact you can produce is a playbook that didn’t exist before you got there. It’s how one-time work turns into compound returns for the org.

03 Built

RAID Sentinel

Shipped 2026 · Live

An AI-powered PMO tool that turns project charters into CEO-ready RAID logs. Two-pass LLM extraction. PMI-standard severity scoring. Source-verified quotes. Built solo, end-to-end.

Try it live ↗ Source on GitHub ↗

The why

Every PM I know spends real time scraping project charters for Risks, Assumptions, Issues, and Dependencies — and most of what they produce isn’t board-ready. RAID extraction is the kind of pattern-recognition work an LLM should handle, but only if the output is grounded in source text and scored to a real standard.

So I built it.

What it does

Upload a project charter PDF. RAID Sentinel runs a two-pass LLM pipeline — first-pass extraction, then a senior-PMO audit pass that catches misses and miscategorizations. Output is a PMI-standard RAID log with executive-ready exports: PDF with charts, CSV, Markdown.

  • Two-pass extract + audit First pass is fast; second pass is the senior reviewer. Catches umbrella outcomes (“the program might slip”) and forces extraction of real underlying drivers.
  • Source quotes mandatory Every RAID item carries a 20–40 word verbatim quote from the charter. No hallucinated risks.
  • PMI-standard scoring Real 5×5 matrix arithmetic, not vibes. Severity reasoning tied back to charter language.
  • BYO API key No data stored. Governance-friendly by design. Charters are sensitive documents.
  • Refine loop Every item has a refine-with-feedback path, because executive judgment still has to override the model.
